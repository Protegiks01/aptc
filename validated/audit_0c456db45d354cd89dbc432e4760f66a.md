# Audit Report

## Title
DKG Verification Accepts Identity Point as Dealt Public Key, Allowing Zero-Entropy Contributions

## Summary
The DKG (Distributed Key Generation) verification in the weighted and unweighted PVSS protocols does not validate that the dealt public key is not the identity point. This allows a malicious validator to submit a transcript with zero entropy that passes all verification checks, violating the fundamental DKG security guarantee that all participants must contribute non-trivial entropy to the shared secret.

## Finding Description

The DKG verification process in Aptos uses PVSS (Publicly Verifiable Secret Sharing) transcripts that are verified through three main checks: Schnorr proof-of-knowledge, low-degree test, and encryption correctness. However, none of these checks explicitly validate that the dealt public key is not the identity point (point at infinity). [1](#0-0) [2](#0-1) 

The verification functions check signatures, polynomial degree, and encryption correctness, but do not validate that `V_hat[W]` (weighted protocol) or `V[n]` (unweighted protocol) - the dealt public key - is not the identity point. This is a critical omission because:

1. **Multi-exponentiation can return identity**: The `g1_multi_exp` and `g2_multi_exp` functions legitimately return the identity point when all scalars are zero or the bases array is empty. [3](#0-2) 

2. **Zero polynomial passes low-degree test**: When a dealer creates a transcript with all polynomial coefficients set to zero, all commitment values become the identity point. The low-degree test checks if the multi-exponentiation equals identity, which it does when all inputs are identity points. [4](#0-3) 

3. **Schnorr PoK accepts zero secret**: A valid proof-of-knowledge can be constructed for secret = 0, where the verification equation `g^s = R * pk^e` becomes `g^r = g^r * identity^e = g^r`, which passes. [5](#0-4) 

4. **Contrast with BLS validation**: Aptos' BLS12-381 public key validation explicitly rejects identity points through subgroup checks, establishing that identity points should not be accepted as valid public keys. [6](#0-5) 

5. **Production usage**: This vulnerability affects the real DKG implementation used for on-chain randomness generation in consensus. [7](#0-6) [8](#0-7) 

**Attack Path**: A Byzantine validator can create a transcript with the zero polynomial (all coefficients = 0), resulting in all commitment values and the dealt public key being the identity point. This transcript passes verification at line 368-374 of `real_dkg/mod.rs` because the underlying PVSS verify function lacks identity point validation.

## Impact Explanation

**Severity: HIGH** (Significant Protocol Violation)

This vulnerability violates the fundamental DKG security guarantee that all participants must contribute non-trivial entropy. The security impacts are:

1. **Protocol Violation**: The DKG verification is accepting invalid transcripts that should be rejected. This is analogous to accepting invalid signatures - it's a protocol-level bug regardless of whether immediate fund loss occurs.

2. **Entropy Degradation**: While a single Byzantine dealer contributing zero may not immediately break security if other dealers are honest, the protocol should not accept such contributions. With threshold-based DKG, if multiple Byzantine validators exploit this, security degrades proportionally.

3. **Cryptographic Best Practice Violation**: Industry-standard cryptographic libraries (including Aptos' own BLS12-381 implementation) explicitly reject identity points as invalid public keys. The DKG verification should enforce the same requirement but fails to do so.

4. **Byzantine Behavior Enablement**: This allows Byzantine validators to exhibit behavior (zero-entropy contributions) that the protocol design should prevent. The lack of validation undermines the protocol's security model.

5. **Downstream Impact**: The DKG output is used for validator consensus randomness generation. Accepting zero-entropy contributions undermines the security assumptions of systems relying on this randomness.

This qualifies as **High Severity** per Aptos bug bounty criteria as a "Significant protocol violation" - the verification logic is accepting inputs it should reject, similar to how accepting invalid signatures would be a protocol violation.

## Likelihood Explanation

**Likelihood: Medium**

The attack requires:
- Validator status in the DKG protocol (any validator can participate)
- Basic understanding of elliptic curve cryptography (creating zero polynomial)
- Ability to construct a valid-appearing transcript

The likelihood is medium because:
1. Byzantine validators are explicitly part of Aptos' threat model (< 1/3 Byzantine assumption)
2. The exploit is straightforward - simply set polynomial coefficients to zero
3. Detection is difficult since the transcript appears cryptographically valid to the verification logic
4. The lack of identity point checks is a clear oversight exploitable by any malicious validator

## Recommendation

Add an explicit check in the verification functions to reject identity points as dealt public keys:

**For weighted_protocol.rs** (after line 309):
```rust
// Validate dealt public key is not identity
if self.V_hat[W].is_identity().into() {
    bail!("Dealt public key cannot be the identity point");
}
```

**For unweighted_protocol.rs** (after line 263):
```rust
// Validate dealt public key is not identity  
if self.V[sc.n].is_identity().into() {
    bail!("Dealt public key cannot be the identity point");
}
```

This mirrors the identity point rejection in BLS12-381 public key validation and enforces the security guarantee that all dealers must contribute non-trivial entropy.

## Proof of Concept

A malicious validator can construct a transcript with zero polynomial by:

1. Setting all polynomial coefficients to 0: `f[i] = 0 for all i`
2. Computing commitments: `V_hat[i] = g_2^0 = identity` for all i
3. Computing ciphertexts: `C[i] = h^0 * ek[i]^r = ek[i]^r` with random r
4. Creating Schnorr PoK for secret 0: `(R = g^r, s = r)`
5. Signing with BLS key

This transcript will pass all three verification checks:
- Schnorr PoK: `g^s = g^r = R * identity^e` ✓
- Low-degree test: `g2_multi_exp([identity,...], coeffs) = identity` ✓  
- Encryption correctness: Pairing equation holds with `v = identity` ✓

The verification in `types/src/dkg/real_dkg/mod.rs:368-374` accepts this malicious transcript because the underlying PVSS `verify()` function lacks identity point validation.

pub fn g1_multi_exp(bases: &[G1Projective], scalars: &[blstrs::Scalar]) -> G1Projective {
    if bases.len() != scalars.len() {
        panic!(
            "blstrs's multiexp has heisenbugs when the # of bases != # of scalars ({} != {})",
            bases.len(),
            scalars.len()
        );
    }

    match bases.len() {
        0 => G1Projective::identity(),
        1 => bases[0].mul(scalars[0]),
        _ => G1Projective::multi_exp(bases, scalars),
    }
}

/// Works around the `blst_hell` bug (see README.md).

        panic!(
            "blstrs's multiexp has heisenbugs when the # of bases != # of scalars ({} != {})",
            bases.len(),
            scalars.len()
        );
    }
    match bases.len() {
        0 => G2Projective::identity(),
        1 => bases[0].mul(scalars[0]),
        _ => G2Projective::multi_exp(bases, scalars),
    }
}

    pub fn low_degree_test_on_g2(self, evals: &Vec<G2Projective>) -> anyhow::Result<()> {
        if evals.len() != self.n {
            bail!("Expected {} evaluations; got {}", self.n, evals.len())
        }

        if self.t == self.n {
            return Ok(());
        }

        let v_times_f = self.dual_code_word();

        debug_assert_eq!(evals.len(), v_times_f.len());
        let zero = g2_multi_exp(evals.as_ref(), v_times_f.as_slice());

        (zero == G2Projective::identity())
            .then_some(())
            .context(format!(
                "the LDT G2 multiexp should return zero, but instead returned {}",
                zero
            ))
    }

pub fn pok_prove<Gr, R>(a: &Scalar, g: &Gr, pk: &Gr, rng: &mut R) -> PoK<Gr>
where
    Gr: Serialize + Group + for<'a> Mul<&'a Scalar, Output = Gr>,
    R: rand_core::RngCore + rand_core::CryptoRng,
{
    debug_assert!(g.mul(a).eq(pk));

    let r = random_scalar(rng);
    let R = g.mul(&r);
    let e = schnorr_hash(Challenge::<Gr> { R, pk: *pk, g: *g });
    let s = r + e * a;

    (R, s)
}

/// Computes the Fiat-Shamir challenge in the Schnorr PoK protocol given an instance $(g, pk = g^a)$
/// and the commitment $R = g^r$.
#[allow(non_snake_case)]
fn schnorr_hash<Gr>(c: Challenge<Gr>) -> Scalar
where
    Gr: Serialize,
{
    let c = signing_message(&c)
        .expect("unexpected error during Schnorr challenge struct serialization");

    hash_to_scalar(&c, SCHNORR_POK_DST)
}

/// Verifies all the $n$ Schnorr PoKs by taking a random linear combination of the verification
/// equations using $(1, \alpha, \alpha^2, \ldots, \alpha^{n-1})$ as the randomness.
///
/// The equation is:
///
///    $$g^{\sum_i s_i \gamma_i} = \prod_i R_i^{\gamma_i} \pk_i^{e_i \gamma_i}$$
///
/// where $e_i$ is the Fiat-Shamir challenge derived by hashing the PK and the generator $g$.
#[allow(non_snake_case)]
pub fn pok_batch_verify<'a, Gr>(
    poks: &Vec<(Gr, PoK<Gr>)>,
    g: &Gr,
    gamma: &Scalar,
) -> anyhow::Result<()>
where
    Gr: Serialize + Group + Mul<&'a Scalar> + HasMultiExp,
{
    let n = poks.len();
    let mut exps = Vec::with_capacity(2 * n + 1);
    let mut bases = Vec::with_capacity(2 * n + 1);

    // Compute \gamma_i = \gamma^i, for all i \in [0, n]
    let mut gammas = Vec::with_capacity(n);
    gammas.push(Scalar::ONE);
    for _ in 0..(n - 1) {
        gammas.push(gammas.last().unwrap().mul(gamma));
    }

    let mut last_exp = Scalar::ZERO;
    for i in 0..n {
        let (pk, (R, s)) = poks[i];

        bases.push(R);
        exps.push(gammas[i]);

        bases.push(pk);
        exps.push(schnorr_hash(Challenge::<Gr> { R, pk, g: *g }) * gammas[i]);

        last_exp += s * gammas[i];
    }

    bases.push(*g);
    exps.push(last_exp.neg());

    if Gr::multi_exp_iter(bases.iter(), exps.iter()) != Gr::identity() {
        bail!("Schnorr PoK batch verification failed");
    }

    Ok(())
}
