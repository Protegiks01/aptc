# Audit Report

## Title
Silent Module Dropping in Bytecode Dependencies Due to BTreeMap Key Collision

## Summary
The `bytecode_deps` field population logic in `compiled_package.rs` silently drops all but one module from multi-module bytecode dependency packages due to using package name as the BTreeMap key for all modules within that package, violating package completeness invariants.

## Finding Description

The vulnerability exists in the package compilation code where bytecode dependencies are loaded. [1](#0-0) 

The code iterates through bytecode dependency packages and their module paths, loading each module via `get_module_in_package()`. However, the critical flaw is that all modules from the same package use the package name as the BTreeMap key: [2](#0-1) 

The `bytecode_deps` field is defined as: [3](#0-2) 

When `try_collect()` collects the iterator of `(PackageName, CompiledModule)` tuples into a `BTreeMap<PackageName, CompiledModule>`, Rust's BTreeMap behavior overwrites earlier values with the same key. For a package like MoveStdlib with multiple modules (vector, signer, string, etc.), all modules map to the same package name key, causing all but the last-inserted module to be silently dropped.

Real-world example: MoveStdlib contains multiple modules: [4](#0-3) 

If loaded as bytecode dependencies, only one module (the last one processed) would be retained in the `bytecode_deps` map.

**Violation of Invariants:**
1. **Deterministic Execution**: Different iteration orders could result in different modules being retained, causing non-deterministic build outputs
2. **Package Completeness**: Packages are intentionally incomplete, missing critical dependency modules
3. **Runtime Correctness**: Code depending on dropped modules will fail at runtime with "module not found" errors

## Impact Explanation

This issue falls into **Medium Severity** because:

1. **State Inconsistencies Requiring Intervention**: Incomplete packages deployed to the blockchain will have missing dependencies, causing transaction failures and requiring redeployment with correct dependencies

2. **Build Reproducibility Failure**: Different build environments or file system orderings could produce different package artifacts, violating deterministic build guarantees

3. **Delayed Error Detection**: The compilation succeeds silently, but failures only manifest at runtime when missing modules are accessed, making debugging extremely difficult

While not directly causing fund loss, this undermines the reliability of the Move package system and could lead to deployed contracts that fail unpredictably, potentially causing service outages or requiring emergency interventions.

## Likelihood Explanation

**High Likelihood** - This occurs automatically whenever:
1. A bytecode dependency package contains multiple modules (common for standard libraries)
2. The package is included in `bytecode_deps` rather than compiled from source
3. No explicit validation checks for package completeness

The MoveStdlib example demonstrates this is not a theoretical edge case - it affects core framework packages that commonly have 10+ modules.

## Recommendation

Change the `bytecode_deps` field type from `BTreeMap<PackageName, CompiledModule>` to `BTreeMap<PackageName, Vec<CompiledModule>>` to support multiple modules per package:

```rust
pub bytecode_deps: BTreeMap<PackageName, Vec<CompiledModule>>,
```

Modify the collection logic to group modules by package:

```rust
bytecode_deps: {
    let mut deps_map: BTreeMap<PackageName, Vec<CompiledModule>> = BTreeMap::new();
    for package in bytecode_deps.iter() {
        let name = package.name.unwrap();
        let modules: Vec<CompiledModule> = package.paths.iter()
            .map(|pkg_path| get_module_in_package(name, pkg_path.as_str()))
            .collect::<Result<Vec<_>>>()?;
        deps_map.insert(name, modules);
    }
    deps_map
}
```

Update all consumers of `bytecode_deps` to handle `Vec<CompiledModule>` instead of single modules.

## Proof of Concept

```rust
#[test]
fn test_bytecode_deps_multi_module_package() {
    use move_symbol_pool::Symbol;
    use std::collections::BTreeMap;
    
    // Simulate a package with 3 modules
    let package_name = Symbol::from("TestPackage");
    let modules = vec![
        ("module_a.mv", create_test_module("TestPackage", "A")),
        ("module_b.mv", create_test_module("TestPackage", "B")),
        ("module_c.mv", create_test_module("TestPackage", "C")),
    ];
    
    // Simulate the current buggy collection logic
    let bytecode_deps: BTreeMap<Symbol, CompiledModule> = modules
        .iter()
        .map(|(_, module)| (package_name, module.clone()))
        .collect();
    
    // Assert the bug: only 1 module retained instead of 3
    assert_eq!(bytecode_deps.len(), 1);
    assert!(bytecode_deps.contains_key(&package_name));
    
    // The correct behavior should retain all 3 modules
    let mut correct_deps: BTreeMap<Symbol, Vec<CompiledModule>> = BTreeMap::new();
    let all_modules: Vec<CompiledModule> = modules.iter().map(|(_, m)| m.clone()).collect();
    correct_deps.insert(package_name, all_modules);
    
    assert_eq!(correct_deps.get(&package_name).unwrap().len(), 3);
}
```

2. Build a project depending on it
3. Inspect the resulting `bytecode_deps` - only one module will be present
4. Attempt to use code from the "missing" modules - runtime failure
