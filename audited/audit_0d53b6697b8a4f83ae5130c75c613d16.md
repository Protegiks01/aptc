# Audit Report

## Title
Unauthenticated Configuration Endpoint Exposes Sensitive Operational Data for Compliance Violations

## Summary
The Aptos Inspection Service exposes a `/configuration` endpoint that returns the entire `NodeConfig` structure without authentication or authorization. While mainnet validators are protected by a config sanitizer, **mainnet fullnodes and API nodes have no such protection**, allowing any network-accessible attacker to retrieve comprehensive operational data including network topology, rate limits, security thresholds, file paths, and service ports. This constitutes a compliance violation under SOC2 (CC6.6, CC6.7) and ISO 27001 (A.9.4.1, A.18.1.3) requirements for production systems.

## Finding Description

The `handle_configuration_request()` function in the Aptos Inspection Service exposes the entire node configuration when the `expose_configuration` flag is enabled. [1](#0-0) 

The inspection service provides no authentication or authorization mechanism, serving requests to any client that can reach the HTTP endpoint. [2](#0-1) 

While private keys are protected via `SilentDebug` formatting, the configuration still exposes extensive operational data:

**Network Configuration Exposure:**
- Seed peer addresses and network topology [3](#0-2) 
- Listen addresses, ports, and connection limits
- Rate limiting configurations with exact thresholds
- Buffer sizes and network tuning parameters

**API Configuration Exposure:**
- TLS certificate and key file paths [4](#0-3) 
- Rate limits for transaction submission and queries
- Maximum gas limits for view functions
- Worker thread counts and runtime configuration

**Consensus Configuration Exposure:**
- Timeout values and backpressure thresholds [5](#0-4) 
- Quorum store parameters
- Block size limits and transaction filtering thresholds

**Critical Gap in Protection:**

The config sanitizer ONLY blocks mainnet validators from exposing configuration: [6](#0-5) 

This means **mainnet fullnodes, VFNs, and API nodes can freely enable this endpoint** without any sanitizer protection. These nodes are frequently operated by commercial services subject to SOC2/ISO 27001 compliance.

Additionally, the sanitizer can be bypassed entirely by setting `skip_config_sanitizer: true`: [7](#0-6) [8](#0-7) 

**Compliance Violations:**

1. **SOC2 CC6.6** (Logical and physical access controls): Exposing network topology, ports, and rate limits provides attackers with a complete infrastructure map
2. **SOC2 CC6.7** (Confidentiality): Security control parameters (rate limits, timeouts, thresholds) should remain confidential
3. **ISO 27001 A.9.4.1** (Information access restriction): No authentication mechanism violates access control requirements
4. **ISO 27001 A.18.1.3** (Protection of records): Configuration data is sensitive operational information requiring protection

## Impact Explanation

This finding is classified as **Low Severity** per the Aptos bug bounty program category "Minor information leaks." While it doesn't directly enable theft of funds, consensus violations, or service disruption, it:

1. **Violates compliance frameworks** required for production blockchain infrastructure services
2. **Aids reconnaissance** for attackers planning more sophisticated attacks
3. **Exposes security controls** allowing attackers to optimize attacks to stay under detection thresholds
4. **Reveals infrastructure topology** useful for network-level attacks
5. **Leaks file system paths** that could aid in other exploitation attempts

Organizations operating production Aptos infrastructure (especially fullnodes and API services) under SOC2 or ISO 27001 certification would face audit findings if this endpoint were enabled.

## Likelihood Explanation

**Medium likelihood** for the following reasons:

1. **Default disabled** on mainnet validators (enforced by sanitizer)
2. **Auto-enabled** on testnet/devnet nodes by the config optimizer [9](#0-8) 
3. **No protection** for mainnet fullnodes, which are commonly operated by commercial services
4. **Requires explicit enabling** for mainnet production systems (but many operators may not realize the compliance implications)
5. **No authentication** means any network-accessible party can retrieve the data

Organizations unfamiliar with compliance requirements may inadvertently enable this for debugging purposes on production systems, creating audit exposure.

## Recommendation

Implement authentication and authorization for the inspection service configuration endpoint:

```rust
// Add to InspectionServiceConfig
pub struct InspectionServiceConfig {
    // ... existing fields ...
    pub configuration_api_key: Option<String>,
}

// Update handle_configuration_request
pub fn handle_configuration_request(
    node_config: &NodeConfig,
    auth_header: Option<&str>,
) -> (StatusCode, Body, String) {
    // Check if endpoint is enabled
    if !node_config.inspection_service.expose_configuration {
        return (
            StatusCode::FORBIDDEN,
            Body::from(CONFIGURATION_DISABLED_MESSAGE),
            CONTENT_TYPE_TEXT.into(),
        );
    }
    
    // Verify authentication if API key is configured
    if let Some(expected_key) = &node_config.inspection_service.configuration_api_key {
        match auth_header {
            Some(header) if header.strip_prefix("Bearer ").map(|t| t == expected_key).unwrap_or(false) => {
                // Authenticated - proceed
            }
            _ => {
                return (
                    StatusCode::UNAUTHORIZED,
                    Body::from("Invalid or missing authentication"),
                    CONTENT_TYPE_TEXT.into(),
                );
            }
        }
    }
    
    let encoded_configuration = format!("{:?}", node_config);
    (StatusCode::OK, Body::from(encoded_configuration), CONTENT_TYPE_TEXT.into())
}
```

Additionally:

1. **Extend sanitizer protection** to all mainnet nodes (not just validators)
2. **Add compliance warnings** to documentation about enabling this endpoint
3. **Implement audit logging** for configuration endpoint access
4. **Consider IP whitelisting** as an additional control layer

## Proof of Concept

**Setup:**
1. Configure a fullnode with `expose_configuration: true`:
```yaml
inspection_service:
  address: "0.0.0.0"
  port: 9101
  expose_configuration: true
```

2. Start the node and access the endpoint:
```bash
curl http://<node-ip>:9101/configuration
```

**Result:**
The endpoint returns the complete `NodeConfig` structure including:
- All network seed peer addresses
- API rate limits and gas limits
- Consensus timeout values
- File system paths for TLS certificates
- Service ports and addresses
- Rate limiting thresholds
- Worker thread counts

**Compliance Audit Scenario:**
During a SOC2 Type II audit, auditors test access controls by:
1. Requesting the configuration endpoint without credentials
2. Observing successful retrieval of operational parameters
3. Documenting violation of CC6.6 (access controls) and CC6.7 (confidentiality)
4. Issuing audit finding requiring remediation before certification

## Notes

While this is correctly classified as **Low severity** per bug bounty standards (minor information leak), it represents a real compliance risk for organizations operating production Aptos infrastructure under SOC2 or ISO 27001 certification. The lack of authentication on the inspection service configuration endpoint, combined with inadequate sanitizer protection for non-validator mainnet nodes, creates audit exposure for commercial blockchain service providers.

The issue is particularly concerning because the config optimizer automatically enables this endpoint on non-mainnet chains, potentially creating a pattern where operators mistakenly enable it on mainnet production systems without understanding the compliance implications.

pub fn handle_configuration_request(node_config: &NodeConfig) -> (StatusCode, Body, String) {
    // Only return configuration if the endpoint is enabled
    let (status_code, body) = if node_config.inspection_service.expose_configuration {
        // We format the configuration using debug formatting. This is important to
        // prevent secret/private keys from being serialized and leaked (i.e.,
        // all secret keys are marked with SilentDisplay and SilentDebug).
        let encoded_configuration = format!("{:?}", node_config);
        (StatusCode::OK, Body::from(encoded_configuration))
    } else {
        (
            StatusCode::FORBIDDEN,
            Body::from(CONFIGURATION_DISABLED_MESSAGE),
        )
    };

    (status_code, body, CONTENT_TYPE_TEXT.into())
}
